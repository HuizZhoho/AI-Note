# 以太网通信协议扩展详解

本文在 `通信协议完整讲解.md` 的基础上继续扩展，以 ZYNQ7010 以太网案例为主线，把工程中经常遇到的以太网相关协议按层次讲清楚。目标不是背协议名，而是知道：

```text
每个协议解决什么问题
它工作在哪一层
它在一次通信中什么时候出现
它和 ZYNQ PS/PL 以太网实现有什么关系
抓包或调试时应该看什么
```

## 1. 协议全景图

以太网系统不是只有 TCP/IP。实际工程里会同时出现这些协议或机制：

```text
物理/接口层：
    10BASE-T / 100BASE-TX / 1000BASE-T
    MII / RMII / RGMII / SGMII
    MDIO / MDC

二层链路层：
    Ethernet II
    IEEE 802.3
    802.1Q VLAN
    802.1p Priority
    802.3x PAUSE
    LLDP
    STP / RSTP / MSTP
    LACP

二层到三层辅助：
    ARP
    IPv6 NDP
    IGMP / MLD

三层网络层：
    IPv4
    IPv6
    ICMP
    ICMPv6

四层传输层：
    UDP
    TCP

常见应用层：
    DHCP
    DNS
    HTTP / HTTPS
    NTP
    SNMP
    MQTT
    Modbus TCP
    RTP/RTSP
    PTP / IEEE 1588
```

在 ZYNQ7010 裸机 lwIP Echo Server 中，最核心的实际链路是：

```text
PHY/RGMII/MDIO
  -> Ethernet
  -> ARP
  -> IPv4
  -> ICMP 或 TCP
  -> TCP Echo 应用
```

如果工程扩展成工业控制、视频传输、网络管理或时间同步，就会引入 UDP、DHCP、DNS、SNMP、PTP、Modbus TCP、MQTT 等协议。

## 2. 物理层和 MAC-PHY 接口

### 2.1 10BASE-T / 100BASE-TX / 1000BASE-T

这些是网线侧的以太网物理层标准。

```text
10BASE-T：
    10 Mbps，双绞线。

100BASE-TX：
    100 Mbps，双绞线。

1000BASE-T：
    1000 Mbps，双绞线，常见千兆以太网。
```

在 ZYNQ7010 上，PS GEM 本身是 MAC，不直接接网线。真正跑这些物理层标准的是外部 PHY 芯片。

```text
ZYNQ GEM
  |
RGMII/RMII/MII
  |
PHY
  |
网络变压器
  |
RJ45
  |
网线
```

调试重点：

```text
PHY link 灯是否亮
PHY 是否完成 auto negotiation
协商速率是否和 GEM 配置一致
PHY 参考时钟是否正常
```

### 2.2 MII / RMII / RGMII / SGMII

这些是 MAC 和 PHY 之间的接口，不是 TCP/IP 协议。

| 接口 | 常见速率 | 特点 | ZYNQ 工程关注点 |
| --- | --- | --- | --- |
| MII | 10/100M | 4bit 数据，TX/RX 时钟 | 引脚多，传统百兆 |
| RMII | 10/100M | 2bit 数据，50 MHz 参考时钟 | 引脚少，常见百兆 |
| RGMII | 10/100/1000M | 4bit DDR，千兆 125 MHz | delay、时序、IO 电压 |
| SGMII | 10/100/1000M | 串行接口 | 常见于高速 PHY/SerDes，ZYNQ7010 PS 侧不如 RGMII 常见 |

RGMII 是 ZYNQ7010 千兆以太网案例最常见接口之一。它的关键点是：

```text
TXD[3:0] 和 RXD[3:0] 是 4bit 数据线
上升沿和下降沿都传输数据
千兆模式下时钟为 125 MHz
TX/RX clock 和 data 之间需要合适 delay
```

RGMII delay 配错时，常见现象是：

```text
PHY link up，但 ping 不通
100M 偶尔通，1000M 不通
抓包看到 CRC 错误
收包不稳定
```

### 2.3 MDIO / MDC

MDIO/MDC 是 PHY 管理接口。

```text
MDC：
    管理时钟，由 MAC 输出。

MDIO：
    双向管理数据线，用于读写 PHY 寄存器。
```

MDIO 用来做：

```text
读取 PHY ID
复位 PHY
启动自协商
读取 link up/down
读取 10/100/1000M 速率
读取全双工/半双工状态
配置部分 PHY 特性
```

在代码调用链里，MDIO 通常藏在 `xemac_add()` 或 GEM 驱动初始化内部：

```text
main()
  |
xemac_add()
  |
xemacpsif_init()
  |
detect_phy()
  |
XEmacPs_PhyRead() / XEmacPs_PhyWrite()
```

调试时如果 PHY ID 读不到，优先查：

```text
PHY 地址
MDIO/MDC 管脚
MDIO 上拉
PHY reset
PHY 电源
PHY 参考时钟
Vivado GEM/MDIO 配置
```

## 3. Ethernet II 和 IEEE 802.3

### 3.1 它解决什么问题

Ethernet 解决的是“局域网里如何把一帧数据从一个 MAC 地址送到另一个 MAC 地址”。

简化以太网帧：

```text
Preamble
SFD
Destination MAC
Source MAC
EtherType / Length
Payload
FCS
```

软件抓包通常从 Destination MAC 开始看不到 Preamble 和 SFD，因为这些通常由 MAC/PHY 硬件处理。

### 3.2 Ethernet II 帧

Ethernet II 使用 EtherType 表示上层协议：

```text
0x0800  IPv4
0x0806  ARP
0x86DD  IPv6
0x8100  802.1Q VLAN
```

ZYNQ GEM 收到一帧后，lwIP 根据 EtherType 判断交给 ARP、IPv4 还是 IPv6 处理。

### 3.3 FCS

FCS 是帧校验序列。它用于检测链路层传输错误。

```text
PHY/MAC 收包
  |
硬件检查 FCS
  |
错误帧通常被丢弃
  |
正确帧才交给 DMA/lwIP
```

如果端口 CRC/FCS 错误增长，通常不是 TCP/IP 软件问题，而是：

```text
网线
PHY
RGMII 时序
双工不匹配
电磁干扰
IO 电压或时钟异常
```

## 4. VLAN 802.1Q 和优先级 802.1p

### 4.1 VLAN 解决什么问题

VLAN 把一个物理二层网络划分成多个逻辑二层广播域。

```text
VLAN 10：控制网络
VLAN 20：视频网络
VLAN 30：管理网络
```

不同 VLAN 默认二层不互通，需要三层网关或路由。

### 4.2 802.1Q Tag

802.1Q 在以太网帧中插入 4 字节 tag：

```text
Destination MAC
Source MAC
TPID = 0x8100
TCI
Original EtherType
Payload
FCS
```

TCI 中包括：

```text
PCP：
    3bit 优先级。

DEI：
    丢弃指示。

VID：
    12bit VLAN ID。
```

### 4.3 ZYNQ 工程中的 VLAN

裸机 lwIP echo 示例通常不启用 VLAN。若要支持 VLAN，需要考虑：

```text
GEM 是否允许接收 VLAN tagged frame
lwIP netif 是否处理 VLAN tag
驱动是否剥离或保留 VLAN tag
交换机端口是 access 还是 trunk
PC 侧网卡是否能打 VLAN tag
```

典型场景：

```text
ZYNQ 接交换机 access 口：
    ZYNQ 收到普通 untagged 帧，最简单。

ZYNQ 接交换机 trunk 口：
    ZYNQ 可能收到 tagged 帧，软件或硬件要能识别 VLAN。
```

### 4.4 优先级

802.1p 使用 VLAN tag 中 PCP 字段表示二层优先级。

```text
0 - 7
```

它不会增加带宽，只是在拥塞时帮助交换机决定哪个队列先发。

## 5. ARP 和 IPv6 NDP

### 5.1 ARP

ARP 用于 IPv4：

```text
已知 IP，寻找 MAC。
```

过程：

```text
PC：谁是 192.168.1.10？
ZYNQ：我是 192.168.1.10，我的 MAC 是 00:0A:35:00:01:02。
```

在 ZYNQ lwIP 中，ARP 通常由协议栈自动处理。应用层不需要手写 ARP。

代码路径：

```text
xemacif_input()
  |
netif->input()
  |
ethernet_input()
  |
etharp_input()
```

### 5.2 IPv6 NDP

IPv6 不使用 ARP，而使用 NDP。NDP 基于 ICMPv6。

常见 NDP 消息：

```text
Neighbor Solicitation
Neighbor Advertisement
Router Solicitation
Router Advertisement
```

ZYNQ7010 裸机 lwIP 示例通常以 IPv4 为主。如果工程要支持 IPv6，需要确认：

```text
lwIP 是否启用 IPv6
netif 是否设置 IPv6 地址
ICMPv6/NDP 是否启用
内存池是否足够
```

## 6. IPv4 / IPv6 / ICMP

### 6.1 IPv4

IPv4 负责主机寻址和路由。

典型配置：

```text
IP:      192.168.1.10
Mask:    255.255.255.0
Gateway: 192.168.1.1
```

在 ZYNQ 代码中：

```c
IP4_ADDR(&ipaddr,  192, 168, 1, 10);
IP4_ADDR(&netmask, 255, 255, 255, 0);
IP4_ADDR(&gw,      192, 168, 1, 1);
```

### 6.2 IPv6

IPv6 使用 128bit 地址，例如：

```text
fe80::1
2001:db8::1234
```

IPv6 工程关注：

```text
NDP 替代 ARP
ICMPv6 是基础协议，不能随便全禁
路由器通告 RA 可用于 SLAAC
路径 MTU 发现依赖 ICMPv6
```

### 6.3 ICMP / ping

ping 对 IPv4 使用 ICMP Echo Request/Reply。

```text
PC -> ZYNQ: ICMP Echo Request
ZYNQ -> PC: ICMP Echo Reply
```

ping 通说明：

```text
PHY link 基本正常
GEM 收发基本正常
ARP 正常
IPv4 正常
ICMP 正常
主循环 xemacif_input() 正常
```

ping 通不代表 TCP 应用正常，因为 TCP 还需要端口、监听、回调和发送缓冲。

## 7. UDP

### 7.1 它解决什么问题

UDP 是无连接传输协议。

特点：

```text
不握手
不保证到达
不保证顺序
不自动重传
开销小
延迟低
```

适合：

```text
设备发现
简单控制
日志上报
实时采样
视频/音频部分场景
自定义嵌入式协议
```

### 7.2 ZYNQ UDP 示例逻辑

裸机 lwIP UDP 应用通常是：

```c
struct udp_pcb *pcb;

pcb = udp_new();
udp_bind(pcb, IP_ADDR_ANY, 5001);
udp_recv(pcb, udp_recv_callback, NULL);
```

接收回调：

```c
static void udp_recv_callback(void *arg,
                              struct udp_pcb *pcb,
                              struct pbuf *p,
                              const ip_addr_t *addr,
                              u16_t port)
{
    if (p == NULL) {
        return;
    }

    udp_sendto(pcb, p, addr, port);
    pbuf_free(p);
}
```

注意：

```text
UDP 是按数据报处理。
一个 udp_sendto() 对应一个 UDP datagram。
相比 TCP，不存在字节流粘包问题。
```

## 8. TCP

### 8.1 它解决什么问题

TCP 提供可靠、有序、面向连接的字节流。

它负责：

```text
三次握手
ACK 确认
序列号
窗口
重传
拥塞控制
四次挥手
```

### 8.2 ZYNQ TCP Echo 逻辑

代码主线：

```text
start_application()
  |
tcp_new_ip_type()
  |
tcp_bind()
  |
tcp_listen()
  |
tcp_accept()
  |
accept_callback()
  |
tcp_recv()
  |
recv_callback()
  |
tcp_recved()
  |
tcp_write()
  |
tcp_output()
```

TCP 是字节流。应用层如果做自定义协议，必须自己设计消息边界：

```text
固定长度
长度字段 + payload
分隔符
TLV
```

### 8.3 常见 TCP 问题

```text
ping 通但 TCP 连不上：
    端口没监听、防火墙、start_application() 没调用、tcp_bind() 失败。

TCP 能连但无回包：
    recv_callback() 未注册、tcp_write() 失败、没有 tcp_output()、pbuf 未释放。

大数据传输卡住：
    tcp_recved() 未调用、lwIP 内存池太小、TCP 窗口太小、cache/DMA 问题。
```

## 9. DHCP / DNS / NTP / SNMP

### 9.1 DHCP

DHCP 自动获取：

```text
IP
Mask
Gateway
DNS
Lease time
```

ZYNQ 裸机调试建议先用静态 IP。DHCP 会增加变量：

```text
是否有 DHCP server
是否收到 Offer
是否租约成功
是否超时回退
```

### 9.2 DNS

DNS 把域名解析成 IP。嵌入式设备如果主动访问服务器，例如 MQTT broker 或 HTTP API，就可能需要 DNS。

流程：

```text
应用请求 example.com
  |
lwIP DNS 查询
  |
拿到 A/AAAA 记录
  |
再建立 TCP/UDP 通信
```

### 9.3 NTP

NTP 用于网络校时，常用 UDP 123。

ZYNQ 场景：

```text
采集设备需要统一时间戳
日志需要准确时间
多设备协同需要大致同步
```

如果需要更高精度，考虑 PTP/IEEE 1588。

### 9.4 SNMP

SNMP 用于网络管理。ZYNQ 作为普通嵌入式节点时不一定需要，但如果设备要被网管平台监控，可以实现 SNMP Agent。

常见监控项：

```text
端口状态
流量计数
错误计数
系统温度
固件版本
运行时间
```

## 10. 工业与应用层协议

### 10.1 Modbus TCP

Modbus TCP 常用于工业控制。

特点：

```text
基于 TCP
默认端口 502
请求/响应模型
报文结构简单
```

ZYNQ 应用：

```text
ARM PS 跑 Modbus TCP Server
PL 采集寄存器或控制 IO
PS 通过 AXI GPIO/BRAM/DMA 读取 PL 数据
网络侧通过 Modbus TCP 暴露寄存器
```

### 10.2 MQTT

MQTT 常用于物联网。

特点：

```text
基于 TCP
发布/订阅模型
需要 broker
适合设备状态和遥测数据
```

ZYNQ 应用：

```text
ZYNQ 采集数据
  |
PS 打包 JSON 或二进制 payload
  |
MQTT publish 到 broker
```

### 10.3 RTP/RTSP

视频流场景可能使用 RTP/RTSP。

```text
RTSP：
    控制会话。

RTP：
    传输媒体数据。
```

如果 PL 侧做视频采集或编码，常见路径是：

```text
PL video pipeline
  |
VDMA / DMA
  |
DDR
  |
PS 网络协议栈发送 RTP/UDP
```

### 10.4 PTP / IEEE 1588

PTP 用于高精度时间同步。

普通 NTP 可以达到毫秒级或更差；PTP 在硬件时间戳支持下可达到更高精度。

ZYNQ 场景：

```text
多设备同步采样
工业实时控制
分布式测量
音视频同步
```

实现关键：

```text
MAC/PHY 是否支持硬件 timestamp
驱动是否暴露 timestamp
软件 PTP 栈是否适配
系统时钟如何校正
```

## 11. 协议选择建议

| 场景 | 推荐协议 | 原因 |
| --- | --- | --- |
| 入门测试 | ICMP ping + TCP echo | 验证链路和协议栈 |
| 简单快速数据 | UDP | 开销小，易实现 |
| 可靠控制 | TCP | 有连接和重传 |
| 工业寄存器访问 | Modbus TCP | 简单、常见 |
| 物联网遥测 | MQTT | 发布/订阅，生态成熟 |
| Web 配置页面 | HTTP/HTTPS | 易调试，用户友好 |
| 自动获取 IP | DHCP | 减少人工配置 |
| 域名访问服务器 | DNS | 服务器 IP 可变 |
| 时间同步 | NTP/PTP | 日志和多设备同步 |
| 网络管理 | SNMP | 兼容网管平台 |

## 12. 抓包过滤速查

```text
eth
vlan
arp
icmp
icmpv6
ip
ipv6
udp
tcp
dhcp
dns
ntp
snmp
tcp.port == 7
udp.port == 5001
tcp.port == 502
tcp.flags.syn == 1
tcp.analysis.retransmission
```

调试顺序：

```text
1. 看 PHY link。
2. 看 ARP/NDP。
3. 看 ICMP ping。
4. 看 TCP/UDP 端口。
5. 看应用层报文。
6. 看重传、丢包、乱序和窗口。
```
