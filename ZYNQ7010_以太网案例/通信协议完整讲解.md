# ZYNQ7010 以太网案例通信协议与电气特性完整讲解

本文专门解释 ZYNQ7010 以太网案例里真正涉及的通信协议、MAC-PHY 电气接口、lwIP 软件抽象，以及从上电到 TCP Echo 数据返回的完整通信交互过程。

先给出一句总线索：

```text
PHY/RGMII/MDIO -> Ethernet -> ARP -> IPv4 -> ICMP/TCP/UDP -> TCP Echo -> lwIP 抽象
```

注意：`RGMII`、`RMII`、`MII` 严格来说不是 TCP/IP 协议，它们是 MAC 和 PHY 之间的数据接口标准；`MDIO/MDC` 是 MAC 管理 PHY 的管理接口。工程学习时必须把这些接口一起理解，因为协议能不能跑起来，首先取决于这些电气连接和时序是否正确。

## 1. 总体分层

可以把整个案例分成两条线：硬件电气链路和软件协议链路。

硬件链路：

```text
PC 网卡
  |
网线 / RJ45
  |
网络变压器
  |
PHY 芯片
  |
RGMII / RMII / MII
  |
ZYNQ7010 PS GEM
  |
DMA
  |
DDR
```

软件协议链路：

```text
Ethernet Frame
  |
ARP / IPv4
  |
ICMP / TCP / UDP
  |
TCP Echo Server
  |
lwIP netif / pbuf / PCB / callback
```

通俗理解：

```text
PHY 负责让网线上的电信号变成芯片能处理的数据。
RGMII/RMII/MII 负责让 PHY 和 ZYNQ 的 GEM 传数据。
MDIO/MDC 负责让 ZYNQ 配置和查询 PHY。
Ethernet 负责局域网内按 MAC 地址收发帧。
ARP 负责把 IP 地址转换成 MAC 地址。
IPv4 负责跨网络寻址。
ICMP 负责 ping 这类网络测试。
TCP 负责可靠连接和数据传输。
UDP 负责简单快速、不保证可靠的数据报传输。
Echo Server 负责收到 TCP 数据后原样返回。
```

## 2. PHY：物理层芯片

### 它解决什么问题

ZYNQ7010 内部有 GEM，也就是以太网 MAC，但没有直接连接网线的模拟物理层。PHY 芯片负责把网线上的差分电信号转换成数字接口信号，再交给 ZYNQ GEM。

PHY 主要做这些事：

```text
电气信号收发
链路检测
10M / 100M / 1000M 自协商
全双工 / 半双工协商
网线极性和状态检测
通过 MDIO 暴露状态寄存器
```

### 在本案例中的位置

```text
RJ45 / 网络变压器 -> PHY -> RGMII/RMII/MII -> ZYNQ GEM
```

PHY 左边接真实网线，右边接 ZYNQ。ZYNQ 软件不能直接感知网线电气细节，它通过 MDIO 读取 PHY 寄存器来判断 link 是否建立、速率是多少。

### 电气关注点

常见工程关注项：

```text
PHY 供电电压是否正确
MAC-PHY IO 电压是否匹配
PHY reset 是否释放
PHY 参考时钟是否存在
RGMII/RMII/MII 接口模式是否和 Vivado 配置一致
MDIO 是否有上拉
PHY 地址 strap 电阻是否和软件匹配
RJ45 网络变压器和中心抽头供电是否正确
```

常见 PHY 侧状态：

```text
Link Down：
    网线未插、对端未开、PHY reset 未释放、时钟异常、接口配置错误。

Link Up：
    PHY 已经检测到对端并完成链路建立。

Auto Negotiation Complete：
    双方完成速率和双工协商。
```

### 代码映射

PHY 初始化通常隐藏在 `xemac_add()` 调用的底层流程里：

```text
main_annotated.c
  |
  |-- lwip_init()
  |
  `-- xemac_add()
        |
        |-- 初始化 GEM
        |-- 通过 MDIO 读取 PHY ID
        |-- 等待 PHY link up
        `-- 根据协商结果设置 GEM speed
```

### 调试验证

优先看这些现象：

```text
PHY link 灯是否亮
串口是否打印 link up
MDIO 是否能读出 PHY ID
Wireshark 是否能看到 ARP 请求/应答
```

如果 PHY ID 读不到，先查 MDIO/MDC、PHY 地址、reset、供电和参考时钟。

## 3. RGMII / RMII / MII：MAC 和 PHY 的数据接口

这一类接口负责在 ZYNQ GEM 和外部 PHY 之间传输真正的以太网数据。

### 3.1 RGMII

RGMII 常用于千兆以太网设计。

典型信号：

```text
TXD[3:0]    MAC -> PHY，发送数据
TX_CTL      MAC -> PHY，发送控制
TXC         MAC -> PHY，发送时钟
RXD[3:0]    PHY -> MAC，接收数据
RX_CTL      PHY -> MAC，接收控制
RXC         PHY -> MAC，接收时钟
```

RGMII 的核心特点：

```text
4bit 数据线
DDR 双沿传输
千兆时典型时钟 125 MHz
上升沿和下降沿都传数据
需要关注时钟和数据之间的 delay
```

通俗理解：

```text
RGMII 线不多，但跑得快。
它用 4 根数据线配合双沿采样，把等效数据吞吐做上去。
```

RGMII delay 是常见坑。因为数据和时钟同时高速变化，接收端需要在合适的时间采样数据。delay 可能由 PHY 内部加，也可能由 PCB 走线或 FPGA/PS 侧配置承担。常见模式：

```text
RGMII：
    不内置 delay。

RGMII-ID：
    TX 和 RX 都由 PHY 或接口提供内部 delay。

RGMII-TXID：
    只给 TX 加 delay。

RGMII-RXID：
    只给 RX 加 delay。
```

如果 delay 配错，常见现象是：

```text
PHY link up，但 ping 不通
偶尔能收到包
大量 CRC 错误
100M 能通，1000M 不通
```

### 3.2 RMII

RMII 常用于 10/100M 以太网。

典型信号：

```text
TXD[1:0]    MAC -> PHY
TX_EN       MAC -> PHY
RXD[1:0]    PHY -> MAC
CRS_DV      PHY -> MAC
REF_CLK     50 MHz 参考时钟
```

RMII 的核心特点：

```text
2bit 数据线
50 MHz 参考时钟
常用于百兆设计
信号数量比 MII 少
```

通俗理解：

```text
RMII 是减少引脚数量的百兆接口。
它用更少的数据线和统一参考时钟完成 MAC-PHY 数据传输。
```

RMII 常见问题：

```text
50 MHz REF_CLK 缺失
REF_CLK 方向配置错误
Vivado 选择了 MIO/EMIO 但硬件实际不匹配
PHY 工作模式 strap 配置不对
```

### 3.3 MII

MII 是较传统的 10/100M MAC-PHY 接口。

典型信号：

```text
TXD[3:0]    MAC -> PHY
TX_EN       MAC -> PHY
TX_CLK      PHY -> MAC 或接口时钟
RXD[3:0]    PHY -> MAC
RX_DV       PHY -> MAC
RX_CLK      PHY -> MAC
```

MII 的核心特点：

```text
4bit 数据线
10/100M
发送和接收时钟相对独立
引脚数量比 RMII 多
```

### 接口选择总结

| 接口 | 常见速率 | 数据线 | 时钟特点 | 常见用途 |
| --- | --- | --- | --- | --- |
| RGMII | 10/100/1000M | TX 4bit + RX 4bit | 千兆 125 MHz，双沿传输 | 千兆以太网 |
| RMII | 10/100M | TX 2bit + RX 2bit | 50 MHz REF_CLK | 少引脚百兆 |
| MII | 10/100M | TX 4bit + RX 4bit | TX/RX 时钟 | 传统百兆 |

## 4. MDIO / MDC：PHY 管理接口

### 它解决什么问题

RGMII/RMII/MII 负责传输数据，MDIO/MDC 负责管理 PHY。

```text
MDC：
    管理时钟，通常由 MAC 输出。

MDIO：
    管理数据线，双向，用于读写 PHY 寄存器。
```

### 在本案例中的作用

ZYNQ 通过 MDIO 做这些事：

```text
读取 PHY ID
判断 PHY 是否存在
配置 PHY reset
启动或查询自动协商
读取 link 状态
读取协商出的速率和双工模式
```

### 电气关注点

通用工程注意事项：

```text
MDIO 通常需要上拉
MDC 频率不能超过 PHY 数据手册限制
PHY 地址由硬件 strap 电阻决定
软件 PHY 地址必须和硬件一致
多个 PHY 可以共享 MDC/MDIO，但地址必须不同
```

### 读写寄存器的理解

MDIO 不是传输网络数据包的，它只是在配置 PHY。可以理解成：

```text
RGMII/RMII/MII：
    货车通道，运输真正的数据包。

MDIO/MDC：
    管理办公室，查询状态和下发配置。
```

### 常见错误

```text
读不到 PHY ID：
    PHY 地址错误、MDIO/MDC 接错、PHY 未供电、reset 未释放、时钟缺失。

能读 PHY ID 但 link 不 up：
    网线/对端问题、PHY 模式错误、参考时钟异常、自协商失败。
```

## 5. Ethernet：以太网帧与 MAC 地址

### 它解决什么问题

Ethernet 负责局域网内按 MAC 地址发送和接收一帧数据。

一帧以太网数据大致是：

```text
目的 MAC
源 MAC
类型 Type
载荷 Payload
FCS 校验
```

常见 Type：

```text
0x0806  ARP
0x0800  IPv4
```

### 在本案例中的位置

```text
PHY -> GEM -> Ethernet Frame -> lwIP
```

GEM 负责收发 Ethernet frame，lwIP 负责看 `Type` 字段，然后决定交给 ARP 还是 IPv4 模块处理。

### MAC 地址

MAC 地址是链路层地址，例如：

```text
00:0A:35:00:01:02
```

它在 `main_annotated.c` 中配置：

```c
unsigned char mac_ethernet_address[] = {
    0x00, 0x0A, 0x35, 0x00, 0x01, 0x02
};
```

注意：

```text
同一局域网内 MAC 地址不能重复。
IP 地址可以改，MAC 地址用于以太网帧真正投递。
```

### 调试验证

Wireshark 中可以看到：

```text
Source      源 MAC
Destination 目的 MAC
Type        ARP 或 IPv4
```

如果 PC 发出了 ARP，但 ZYNQ 没有回应，问题可能在：

```text
PHY/GEM 收包
MAC 地址过滤
IP 配置
lwIP 主循环
xemacif_input()
```

## 6. ARP：把 IP 地址转换成 MAC 地址

### 它解决什么问题

PC 知道 ZYNQ 的 IP，比如：

```text
192.168.1.10
```

但以太网帧真正发送时需要目的 MAC。ARP 的作用就是问：

```text
谁是 192.168.1.10？请告诉我你的 MAC 地址。
```

ZYNQ 回答：

```text
192.168.1.10 在我这里，我的 MAC 是 00:0A:35:00:01:02。
```

### 完整过程

```text
PC 发送 ARP Request，目的 MAC 是广播 ff:ff:ff:ff:ff:ff
  |
ZYNQ 收到广播帧
  |
lwIP 判断目标 IP 是自己
  |
ZYNQ 发送 ARP Reply
  |
PC 把 IP-MAC 关系放入 ARP 缓存
```

### 代码映射

ARP 由 lwIP 内部处理，主线仍然是：

```text
xemacif_input()
  |
netif->input()
  |
lwIP ARP module
```

用户代码通常不需要手写 ARP。

### 调试验证

PC 端查看 ARP 缓存：

```powershell
arp -a
```

Wireshark 过滤：

```text
arp
```

常见现象：

```text
只看到 ARP Request，没有 ARP Reply：
    ZYNQ 没收到包，或 lwIP 没处理，或 IP 不匹配。

有 ARP Reply，但 ping 不通：
    链路层基本通了，继续查 IPv4/ICMP。
```

## 7. IPv4：网络层地址

### 它解决什么问题

IPv4 用 IP 地址标识网络中的主机。

本案例常见配置：

```text
ZYNQ IP: 192.168.1.10
PC IP:   192.168.1.100
Mask:    255.255.255.0
Gateway: 192.168.1.1
```

在同一网段内，PC 可以直接通过 ARP 找到 ZYNQ 的 MAC。

### 在本案例中的位置

```text
Ethernet Type 0x0800
  |
IPv4
  |
ICMP / TCP / UDP
```

IPv4 包头里有一个 Protocol 字段：

```text
1   ICMP
6   TCP
17  UDP
```

### 代码映射

静态 IP 在 `main_annotated.c` 中设置：

```c
IP4_ADDR(&ipaddr,  192, 168, 1, 10);
IP4_ADDR(&netmask, 255, 255, 255, 0);
IP4_ADDR(&gw,      192, 168, 1, 1);
```

然后通过 `xemac_add()` 绑定到 `netif`。

### 常见错误

```text
PC 和 ZYNQ 不在同一网段
PC 网卡选错
ZYNQ IP 和局域网其他设备冲突
网关配置误解
```

同一根网线直连测试时，网关可以不是重点；IP 和 mask 同网段才是重点。

## 8. ICMP / ping：网络连通性测试

### 它解决什么问题

ping 使用 ICMP Echo Request / Echo Reply 来测试网络是否可达。

完整过程：

```text
PC -> ZYNQ: ICMP Echo Request
ZYNQ -> PC: ICMP Echo Reply
```

但第一次 ping 通常先发生 ARP：

```text
ARP Request
ARP Reply
ICMP Echo Request
ICMP Echo Reply
```

### 在本案例中的意义

ping 能通，说明这些部分基本正常：

```text
PHY link
MAC-PHY 接口
GEM 收发
DMA buffer
Ethernet
ARP
IPv4
ICMP
lwIP 主循环
```

ping 通不代表 TCP echo 一定通，因为 TCP 还涉及连接、端口、回调和发送缓冲。

### 调试验证

PC 端：

```powershell
ping 192.168.1.10
```

Wireshark 过滤：

```text
icmp
```

常见现象：

```text
ARP 不通：
    先查底层链路和 IP。

ARP 通，ICMP 不通：
    查 lwIP ICMP 配置、防火墙、netif 状态。
```

## 9. UDP：无连接数据报

### 它解决什么问题

UDP 是无连接传输协议。它不建立连接，不保证可靠到达，不保证顺序。

特点：

```text
开销小
速度快
实现简单
不保证可靠
需要应用层自己处理丢包和重传
```

### 在本案例中的位置

当前 TCP Echo Server 主线不依赖 UDP，但 lwIP 协议栈通常会支持 UDP。很多嵌入式网络案例会用 UDP 做：

```text
简单数据采集
广播发现设备
低延迟控制
日志发送
```

### 和 TCP 的区别

```text
UDP：
    像寄明信片，发出去就不管。

TCP：
    像打电话，先建立连接，再确认每段数据是否收到。
```

### 调试验证

Wireshark 过滤：

```text
udp
```

如果本工程没有 UDP 应用代码，抓不到 UDP 是正常的。

## 10. TCP：可靠连接与 Echo Server 的核心

### 它解决什么问题

TCP 提供可靠、按顺序、面向连接的数据传输。

它负责：

```text
三次握手建立连接
ACK 确认
窗口控制
丢包重传
顺序恢复
连接关闭
```

### 三次握手

PC 连接 ZYNQ 的 TCP echo 端口时，会发生：

```text
PC   -> ZYNQ: SYN
ZYNQ -> PC:   SYN ACK
PC   -> ZYNQ: ACK
```

通俗理解：

```text
PC：我想和你建立连接。
ZYNQ：可以，我也准备好了。
PC：收到，我们开始传数据。
```

### ACK、窗口、重传

TCP 不是只把数据发出去，它还要确认对方是否收到。

```text
ACK：
    告诉对方，我已经收到哪些字节。

窗口：
    告诉对方，我还能接收多少数据。

重传：
    如果一段数据迟迟没有 ACK，TCP 会再次发送。
```

在 lwIP 应用代码中，`tcp_recved()` 很重要：

```text
它告诉 lwIP：应用已经处理了这些数据，可以释放接收窗口。
```

如果忘记调用，长时间通信可能卡住。

### 代码映射

TCP Echo Server 在 `echo_annotated.c`：

```text
tcp_new_ip_type()
    创建 TCP PCB。

tcp_bind()
    绑定端口 7。

tcp_listen()
    进入监听状态。

tcp_accept()
    注册连接建立回调。

tcp_recv()
    注册数据接收回调。

tcp_recved()
    更新 TCP 接收窗口。

tcp_write()
    把收到的数据写回发送缓冲区。

tcp_output()
    推动数据尽快发出。
```

### TCP Echo 完整过程

```text
PC 建立 TCP 连接
  |
lwIP 调用 accept_callback()
  |
accept_callback() 注册 recv_callback()
  |
PC 发送应用数据
  |
lwIP 调用 recv_callback()
  |
recv_callback() 调用 tcp_recved()
  |
recv_callback() 调用 tcp_write()
  |
ZYNQ 原样返回数据
```

### 调试验证

PC 端连接：

```powershell
telnet 192.168.1.10 7
```

或使用网络调试助手：

```text
协议：TCP Client
目标 IP：192.168.1.10
端口：7
```

Wireshark 过滤：

```text
tcp.port == 7
```

常见现象：

```text
ping 通但 TCP 连不上：
    查端口、start_application()、tcp_bind()、tcp_listen()。

TCP 能连接但不回数据：
    查 recv_callback()、tcp_write()、tcp_output()、pbuf_free()。

传大数据卡住：
    查 tcp_recved()、TCP 窗口、lwIP 内存池。
```

## 11. DHCP：动态获取 IP

### 它解决什么问题

DHCP 用来自动分配 IP 地址。设备上电后向网络中请求地址，路由器或 DHCP 服务器分配 IP。

### 在本案例中的关系

本案例通常使用静态 IP：

```text
ZYNQ IP 固定为 192.168.1.10
PC 手动配置到同一网段
```

如果启用 DHCP，流程会变成：

```text
ZYNQ 广播 DHCP Discover
DHCP Server 回复 Offer
ZYNQ 发送 Request
DHCP Server 发送 ACK
ZYNQ 获得 IP
```

### 调试建议

入门调试建议先用静态 IP，因为变量更少。等 ping 和 TCP echo 稳定后，再考虑启用 DHCP。

## 12. lwIP 抽象：netif、pbuf、PCB、callback

### netif

`netif` 是 lwIP 的网卡对象。

它保存：

```text
IP 地址
子网掩码
网关
MAC 地址
底层发送函数
输入处理函数
网卡状态
```

代码主线：

```text
xemac_add()
  |
创建并注册 netif
```

### pbuf

`pbuf` 是 lwIP 的数据包缓冲区。

接收方向：

```text
DMA buffer -> pbuf -> lwIP -> 应用回调
```

发送方向：

```text
应用数据 -> pbuf / TCP send buffer -> GEM DMA -> PHY
```

`pbuf_free()` 必须正确调用，否则内存池会耗尽。

### PCB

PCB 是协议控制块。TCP PCB 保存 TCP 连接状态。

它保存：

```text
本地端口
远端 IP 和端口
连接状态
窗口大小
回调函数
发送和接收状态
```

### callback

lwIP RAW API 主要靠回调驱动。

```text
有新连接：
    调用 accept_callback()

收到 TCP 数据：
    调用 recv_callback()

连接关闭：
    p == NULL 通知应用
```

通俗理解：

```text
裸机里没有网络线程。
你在 main() 的 while(1) 里不断喂数据给 lwIP。
lwIP 解析出事件后，通过 callback 通知你的应用代码。
```

## 13. 完整通信交互过程

这一节把所有协议串起来。

### 13.1 上电与 PHY reset

```text
板子上电
  |
电源稳定
  |
PHY reset 释放
  |
PHY 加载 strap 配置
  |
PHY 工作在 RGMII/RMII/MII 模式
```

此时需要确认：

```text
PHY 供电正常
参考时钟正常
reset 极性和时序正确
接口模式正确
```

### 13.2 软件初始化

```text
main()
  |
init_platform()
  |
设置 IP / MAC
  |
lwip_init()
  |
xemac_add()
```

`xemac_add()` 内部完成：

```text
初始化 GEM
初始化 DMA
通过 MDIO 查找 PHY
读取 PHY ID
等待 link up
设置 GEM 速率
注册 lwIP netif
```

### 13.3 PHY 自协商

```text
PHY 和 PC 网卡互相声明能力
  |
协商 10M/100M/1000M
  |
协商全双工/半双工
  |
link up
  |
ZYNQ 通过 MDIO 读取结果
  |
GEM 配成相同速率
```

如果 PHY 是 1000M，但 GEM 按 100M 配，数据会异常。

### 13.4 ARP

```text
PC 想 ping 192.168.1.10
  |
PC 不知道 192.168.1.10 的 MAC
  |
PC 广播 ARP Request
  |
ZYNQ lwIP 回 ARP Reply
  |
PC 获得 ZYNQ MAC
```

### 13.5 ICMP ping

```text
PC 发送 ICMP Echo Request
  |
PHY 收到电信号
  |
GEM 收到 Ethernet frame
  |
DMA 写入 DDR
  |
xemacif_input() 取包
  |
lwIP 解析 IPv4 + ICMP
  |
lwIP 生成 ICMP Echo Reply
  |
GEM/DMA/PHY 发回 PC
```

### 13.6 TCP 三次握手

```text
PC -> ZYNQ: SYN
ZYNQ -> PC: SYN ACK
PC -> ZYNQ: ACK
```

完成后，lwIP 调用 `accept_callback()`，应用层知道有一个新连接。

### 13.7 TCP Echo 数据收发

```text
PC 发送数据 "hello"
  |
ZYNQ GEM 收包
  |
lwIP 解析 TCP
  |
调用 recv_callback()
  |
tcp_recved() 更新接收窗口
  |
tcp_write() 写回 "hello"
  |
tcp_output() 推动发送
  |
PC 收到 "hello"
```

这就是本案例的完整通信闭环。

## 15. 调试检查表

建议按这个顺序排查：

```text
1. 串口是否正常打印
2. PHY 供电是否正常
3. PHY reset 是否释放
4. PHY 参考时钟是否存在
5. MDIO 是否能读到 PHY ID
6. PHY link 是否 up
7. 速率和双工是否协商成功
8. Vivado GEM0/GEM1 是否选对
9. RGMII/RMII/MII 是否和原理图一致
10. RGMII delay 是否正确
11. PC 和 ZYNQ 是否同网段
12. Wireshark 是否看到 ARP Request
13. ZYNQ 是否回复 ARP Reply
14. ping 是否有 ICMP Request/Reply
15. TCP 是否完成三次握手
16. recv_callback() 是否进入
17. tcp_write() 是否成功
18. pbuf_free() 是否调用
```

按层次定位：

```text
PHY 灯不亮：
    先查电气、网线、PHY reset、参考时钟。

PHY ID 读不到：
    查 MDIO/MDC、PHY 地址、reset、供电。

ARP 没回应：
    查 GEM 收包、lwIP 主循环、IP 配置。

ping 通但 TCP 不通：
    查 start_application()、端口、TCP 回调。

TCP 能连但不回数据：
    查 recv_callback()、tcp_write()、tcp_output()。
```

## 16. 学习总结

ZYNQ7010 以太网案例不是单个协议在工作，而是一整条链路协同：

```text
电气层保证线能通
PHY 保证网线信号能被芯片理解
RGMII/RMII/MII 保证 PHY 和 GEM 能传帧
MDIO 保证软件能配置和读取 PHY
Ethernet 保证局域网帧能按 MAC 投递
ARP 保证 IP 能找到 MAC
IPv4 保证网络层寻址
ICMP 用于 ping 测试
TCP 保证可靠连接和数据传输
Echo Server 验证应用层收发
lwIP 把这些协议组织成可调用的软件栈
```

掌握这条链路后，再看 `main_annotated.c` 和 `echo_annotated.c`，就能把“协议原理”和“代码调用”对应起来。
