## 一句话概括

Superpowers 是一套面向编程代理的“工程工作流技能包”。它不是某个具体语言、框架或 API 的知识库，而是把需求澄清、方案设计、计划拆解、TDD、系统化调试、代码审查和交付收尾这些工程纪律，包装成一组可被 Codex、Claude Code、Gemini CLI、OpenCode、Cursor 等工具加载的 skills。

它的核心目标是让代理不要一上来就写代码，而是先确认意图、设计方案、拆解计划、隔离工作区、按测试驱动实现、验证结果，再交付。

## 项目定位

仓库 README 对 Superpowers 的定位非常明确：它是一套“agentic skills framework & software development methodology”。换句话说，它解决的不是“代理不会写代码”的问题，而是“代理写代码时太容易跳步骤、过度自信、缺少验证、没有计划、调试靠猜”的问题。

它试图把资深工程师日常会做的流程固化下来：

- 需求不清时先问问题。
- 功能开始前先设计。
- 多步骤任务先写计划。
- 实现前先建立隔离分支或工作区。
- 新功能和 bugfix 优先走 TDD。
- 遇到 bug 先找根因，不靠猜。
- 完成前必须运行验证命令。
- 合并或收尾前做代码审查和工作区清理。

这些规则对小任务可能显得重，但对复杂功能、长期项目和代理自动化开发很有价值。

## 仓库结构解读

仓库同时支持多个 agent harness：

- `.codex-plugin/`：Codex 插件元数据。
- `.claude-plugin/`：Claude Code 插件元数据。
- `.cursor-plugin/`：Cursor 插件相关内容。
- `.opencode/`：OpenCode 插件入口。
- `skills/`：核心技能库，是这个仓库最重要的部分。
- `docs/`：设计文档、计划文档、OpenCode 文档和测试说明。
- `hooks/`：平台相关 hook。
- `assets/`：图标和展示资源。

`package.json` 很轻量，只声明项目名、版本和 OpenCode 插件入口。真正的能力主要在 `skills/` 和各平台插件 manifest 中。

## Codex 插件信息

`.codex-plugin/plugin.json` 声明的插件名是 `superpowers`，版本是 `5.1.0`。描述是：

> planning, TDD, debugging, and collaboration workflows

插件分类是 `Coding`，能力声明包括：

- `Interactive`
- `Read`
- `Write`

它把技能入口指向：

```json
"skills": "./skills/"
```

这意味着 Superpowers 的核心不是二进制程序，而是一组可组合的 `SKILL.md` 文档。Codex App 的官方推荐安装方式是通过插件市场安装；如果直接把 `skills/` 下的目录安装到用户级 skills 目录，也能让这些技能被 skill discovery 发现。

## 整体工作流

Superpowers 的理想工作流大致是：

1. `using-superpowers`：先确认该使用哪些技能。
2. `brainstorming`：把模糊想法澄清成可执行设计。
3. `writing-plans`：把设计拆成细粒度执行计划。
4. `using-git-worktrees`：创建隔离工作区或分支。
5. `executing-plans` 或 `subagent-driven-development`：按计划实现。
6. `test-driven-development`：对功能和 bugfix 走红绿重构。
7. `systematic-debugging`：遇到失败时先找根因。
8. `requesting-code-review`：完成阶段请求审查。
9. `receiving-code-review`：理性处理审查反馈。
10. `verification-before-completion`：交付前必须跑验证。
11. `finishing-a-development-branch`：决定合并、PR、保留或清理。

下面逐项解释每个 skill。

## 1. using-superpowers

用途：这是 Superpowers 的元技能，负责建立“先检查技能，再执行任务”的基本规则。

它要求代理在每次任务开始时判断是否有相关 skill 适用。如果有，即使只有很小概率，也应该先加载相关 skill，再决定如何执行。

它解决的问题：

- 代理凭记忆工作，忽略已安装技能。
- 代理先探索、先改文件、先写答案，事后才想起流程。
- 对简单任务过度自信，跳过必要检查。

实际意义：

这是整个 Superpowers 的入口。如果没有它，其他 skill 可能只在用户明确点名时才被使用；有了它，代理会更主动地把任务映射到对应工作流。

注意：

它的措辞很强，目的是约束代理行为。但在真实 Codex 环境里，系统指令、开发者指令和用户明确指令仍然有更高优先级。

## 2. brainstorming

用途：在做创造性工作前使用，例如新增功能、设计组件、改行为、搭建系统或实现模糊需求。

核心思想：

不要直接写代码，先弄清楚用户真正想要什么。它会推动代理通过问题、备选方案和阶段性确认，把一个粗略想法变成清晰设计。

它解决的问题：

- 用户一句“做个功能”，代理就直接动手。
- 需求边界不清，最后返工。
- 没有设计权衡，代码实现和用户预期错位。
- 小任务被误判为“不需要设计”。

适用场景：

- “我想做一个 XX 功能。”
- “帮我改一下用户体验。”
- “加一个新模块。”
- “这个产品流程要优化。”

不适合：

- 用户已经给了非常明确的 bug 和修复方向。
- 单纯回答事实性问题。
- 已经有完整需求文档，只需要执行。

## 3. writing-plans

用途：当已经有需求或设计，需要把它拆成可执行计划时使用。

核心思想：

计划应该细到一个初级工程师也能照着做，并且每个任务都包含明确文件路径、具体变更和验证方式。

它强调：

- 任务要足够小。
- 不要写占位符式计划。
- 每个任务要能独立验证。
- 计划要先自检再执行。

它解决的问题：

- 计划太大，执行时失控。
- 只有“实现功能 A”这种空泛步骤。
- 写代码时不断重新设计。
- 没有验收标准。

适用场景：

- 多文件修改。
- 新功能分阶段实现。
- 复杂重构。
- 需要交给代理或子代理执行的工作。

## 4. using-git-worktrees

用途：开始需要隔离的功能开发，或执行实现计划前使用。

核心思想：

不要在当前工作区直接做大规模修改。优先检测是否已有隔离环境；如果没有，就通过平台原生能力或 `git worktree` 创建隔离工作区。

它关注：

- 当前是否已经在 worktree 或隔离工作区。
- 分支名如何命名。
- 工作区放在哪里。
- 项目依赖如何初始化。
- 开始前测试基线是否干净。

它解决的问题：

- 在脏工作区里开发，混入无关改动。
- 大功能失败后难以回滚。
- 当前分支已有用户改动，代理误改或覆盖。
- 不验证初始测试状态，最后无法判断失败来自哪里。

适用场景：

- 新功能开发。
- 大型 bugfix。
- 计划驱动的多步骤实现。
- 用户要求不要影响当前工作树。

## 5. executing-plans

用途：已有书面实现计划，并准备按计划执行时使用。

核心思想：

先加载计划并确认，再按任务执行；执行过程中设置 review checkpoint，必要时停止并请求帮助。

它强调：

- 不要跳过计划审查。
- 不要随意偏离计划。
- 每个任务完成后要有验证。
- 遇到不明确、计划错误或环境阻塞时停下来。

它解决的问题：

- 代理拿到计划后只挑容易的做。
- 计划执行中途变成自由发挥。
- 做完没有阶段性验证。
- 复杂任务一路做到最后才发现方向错。

适用场景：

- 用户提供了计划。
- 前一步 `writing-plans` 已经生成计划。
- 需要在一个连续会话中落地多个任务。

## 6. subagent-driven-development

用途：当前会话中执行包含多个独立任务的实现计划。

核心思想：

把实现任务分派给子代理，每个子代理处理一个相对独立的任务；主代理负责协调、审查和整合。

它强调：

- 任务必须尽量独立。
- 子代理 prompt 要包含明确上下文和验收标准。
- 子代理完成后不能盲信，要审查。
- 适合并行或半并行推进复杂计划。

它解决的问题：

- 一个代理长时间处理太多上下文，注意力下降。
- 多个独立任务串行执行效率低。
- 实现和审查混在一起，缺少复核。

限制：

这个 skill 的效果取决于当前平台是否支持子代理或等价能力。如果平台没有真正的子代理工具，只能把它当作任务拆分和审查方法论使用。

## 7. dispatching-parallel-agents

用途：当有两个以上互不依赖的任务，可以并行推进时使用。

核心思想：

先识别独立域，再为每个域创建聚焦任务，并行分派，最后统一 review 和 integration。

它适合：

- 多个独立 bug。
- 多个互不影响的文件或模块。
- 文档、测试、实现可以拆开并行。
- 多个方案调研。

不适合：

- 任务之间有严格顺序依赖。
- 多个任务会同时修改同一核心文件。
- 缺少清晰验收标准。
- 上下文必须共享且无法切分。

它解决的问题：

- 明明可以并行却串行浪费时间。
- 并行任务边界不清导致冲突。
- 子任务没有统一审查，合并质量不稳定。

## 8. test-driven-development

用途：实现功能或修 bug 前使用。

核心思想：

严格执行 RED-GREEN-REFACTOR：

1. RED：先写失败测试。
2. 验证 RED：确认测试真的失败。
3. GREEN：写最小实现让测试通过。
4. 验证 GREEN：确认测试通过。
5. REFACTOR：清理实现，保持测试通过。

它解决的问题：

- 先写实现，再补测试，测试只验证已知实现。
- bugfix 没有回归测试。
- 功能是否成功只靠肉眼判断。
- 代理写了过多代码，不知道哪些是必要的。

适用场景：

- 新功能。
- bugfix。
- 行为变更。
- 有明确输入输出或验收标准的逻辑。

不适合或需调整：

- 纯探索原型。
- UI 视觉微调且测试基础薄弱。
- 外部系统强依赖、短期难以建立自动化测试的任务。

即便不完全执行 TDD，这个 skill 也会提醒代理至少先定义可验证行为。

## 9. systematic-debugging

用途：遇到 bug、测试失败、异常行为时使用。

核心思想：

先找根因，再修复。不要猜，不要随机改，不要只处理表面症状。

它的四阶段思路是：

1. 根因调查：收集证据，复现问题，定位失败点。
2. 模式分析：判断是否有同类问题或系统性原因。
3. 假设和测试：提出假设，用最小实验验证。
4. 实现修复：基于已验证根因修改。

它解决的问题：

- 看见错误就改最近的代码。
- 修了一个症状，根因还在。
- 没有复现路径。
- 没有证明修复有效。
- 引入更多副作用。

适用场景：

- 测试失败。
- 线上/本地 bug。
- 性能异常。
- 偶发问题。
- 用户报告“看起来不对”。

## 10. requesting-code-review

用途：完成任务、大功能或合并前请求代码审查。

核心思想：

在把工作标记完成前，让另一个审查视角检查实现是否满足需求、是否有遗漏、是否有质量问题。

它关注：

- 工作是否满足原需求。
- 是否存在明显 bug。
- 测试是否覆盖关键行为。
- 是否有过度设计。
- 是否有未处理边界条件。

它解决的问题：

- 代理自评过度乐观。
- 完成前没有独立审查。
- 合并后才发现明显问题。

适用场景：

- 大功能完成后。
- 多文件修改后。
- 准备 PR 或 merge 前。
- 用户明确要求 review。

## 11. receiving-code-review

用途：收到代码审查反馈后使用。

核心思想：

不要盲目接受，也不要防御性拒绝。先判断反馈是否正确、是否清晰、是否必要，再决定修改、澄清或反驳。

它强调：

- 技术事实优先。
- 不要表演式同意。
- 不清楚的反馈要追问。
- 对“看起来更专业”的建议做 YAGNI 检查。
- 如果反馈错误，可以礼貌说明理由。

它解决的问题：

- 审查者说什么就改什么。
- 为了“专业”加入不需要的复杂度。
- 错误反馈被实现成错误代码。
- 反馈处理没有优先级。

适用场景：

- 人类 reviewer 给出建议。
- 外部自动 review 工具提出问题。
- 另一个代理审查了代码。
- PR review 需要逐条处理。

## 12. verification-before-completion

用途：在声称“完成了”“修好了”“测试通过了”之前使用。

核心思想：

证据先于结论。不能因为代码看起来对，就说完成；必须实际运行验证命令并确认输出。

它要求：

- 明确运行了什么命令。
- 确认命令成功。
- 如果没有运行，必须说明原因。
- 不要把“应该通过”说成“已经通过”。

它解决的问题：

- 代理没有跑测试却声称测试通过。
- 忽略 lint、build 或类型检查。
- 最终交付缺少证据。
- 用户拿到代码后才发现失败。

适用场景：

- 准备最终回复。
- 准备 commit。
- 准备 PR。
- 修复 bug 后。
- 声称任何成功状态前。

这是最实用的 skill 之一，因为它直接降低“假完成”的风险。

## 13. finishing-a-development-branch

用途：实现完成、测试通过后，决定如何处理开发分支或工作区。

核心思想：

工作完成不等于流程结束。还需要决定是否合并、开 PR、保留分支、清理 worktree，或者放弃修改。

它的步骤包括：

- 再次验证测试。
- 判断当前环境。
- 找到 base branch。
- 向用户展示选项。
- 根据选择合并、推送、开 PR 或清理。
- 清理前确保合并或保存已经成功。

它解决的问题：

- 做完代码后工作区残留。
- 未确认就删除分支或 worktree。
- 没有给用户选择权。
- 合并前没有最后验证。

适用场景：

- 开发分支收尾。
- worktree 开发完成。
- 准备合并到主分支。
- 准备发 PR。

## 14. writing-skills

用途：创建、修改或验证新的 skills。

核心思想：

把 skill 本身也当作工程产物来开发：需要清晰触发条件、精简内容、可测试、能防止代理绕过规则。

它解释了：

- 什么是 skill。
- skill 类型：技术流程、模式、参考资料等。
- `SKILL.md` 的结构。
- description 如何写得适合被代理检索。
- 如何用 TDD 思路测试 skill。
- 如何减少 token 占用。
- 如何避免写成冗长教程。

它解决的问题：

- skill 描述太模糊，代理无法触发。
- skill 内容太长，加载成本高。
- skill 只讲理念，不约束行为。
- skill 没有测试，无法确认是否有效。

适用场景：

- 你要为团队写自己的 Codex/Claude skill。
- 修改已有 skill。
- 把团队流程沉淀成可复用代理能力。

## 这套技能的工程价值

Superpowers 的价值不在于某个单独 skill，而在于它把软件开发流程串成一条完整链路：

- 从“想法”到“设计”：`brainstorming`
- 从“设计”到“计划”：`writing-plans`
- 从“计划”到“隔离环境”：`using-git-worktrees`
- 从“计划”到“实现”：`executing-plans`、`subagent-driven-development`
- 从“实现”到“质量保证”：`test-driven-development`、`systematic-debugging`
- 从“完成”到“可信交付”：`requesting-code-review`、`verification-before-completion`
- 从“交付”到“合并/清理”：`finishing-a-development-branch`

这套链路的本质是减少代理开发中最常见的风险：

- 需求误解。
- 过早实现。
- 缺少计划。
- 工作区污染。
- 测试缺失。
- 调试靠猜。
- 未经审查。
- 未经验证就声称完成。

## 适合的使用场景

Superpowers 适合：

- 长期维护的真实代码库。
- 多文件、多模块、多步骤任务。
- 需要稳定交付的功能开发。
- 重构、修 bug、测试补齐。
- 希望 Codex 更像工程团队成员而不是代码补全工具。
- 想把团队工程规范沉淀为代理工作流。

## 不一定适合的场景

它不一定适合：

- 只问一个简单事实。
- 临时脚本或一次性实验。
- 用户明确要求“不要规划，直接给答案”。
- 没有测试体系且任务极小的项目。
- 只做非工程类文本处理。

即使如此，其中的 `verification-before-completion` 和 `systematic-debugging` 仍然有广泛价值。

## 使用建议

如果你在 Codex 中使用 Superpowers，可以这样和代理配合：

1. 新功能开始时，让它先做 `brainstorming`。
2. 设计确定后，让它输出 `writing-plans` 风格的实现计划。
3. 多步骤实现前要求创建隔离分支或 worktree。
4. 对行为变更明确要求走 TDD。
5. 遇到测试失败时，要求先做 systematic debugging，不要猜修。
6. 最终回复前要求列出实际运行过的验证命令。
7. 合并前要求做 code review。

示例提示：

```text
请按 Superpowers 工作流处理这个功能：先澄清需求和设计，不要直接写代码。
```

```text
已有设计如下，请先写实现计划，每个任务要包含文件路径、具体改动和验证命令。
```

```text
这个测试失败了，请按 systematic-debugging 找根因，不要直接猜修。
```

```text
在说完成前，请运行 verification-before-completion，列出实际验证结果。
```

## 总结

Superpowers 更像是一套“代理工程纪律包”。它不替代程序员判断，也不保证每次都更快；它的优势是降低代理在复杂任务中常见的失误概率，让 Codex 在真实工程场景里更倾向于先理解、再计划、再实现、再验证。

如果你的目标是让 Codex 长时间处理真实项目，而不是只回答零散问题，这套 skills 很值得保留。它最有价值的部分是三个习惯：先澄清、按测试和证据工作、完成前验证。
