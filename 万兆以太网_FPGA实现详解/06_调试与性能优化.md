# 06 调试与性能优化

## 本章解决什么问题

10GbE 调试要同时看：

```text
光模块和物理链路
GT 状态
PCS block lock
MAC 统计
AXI-Stream 握手
DMA descriptor
DDR/cache
Firmware 中断和 buffer
上位机抓包
```

不要只盯应用层。很多问题发生在 GT、PCS、MAC 或 DMA。

## link 不 up

排查顺序：

```text
1. SFP+ 模块是否插入。
2. RX_LOS 是否有效。
3. TX_DISABLE 是否关闭。
4. 对端速率是否 10G。
5. GT refclk 是否正确。
6. QPLL/CPLL 是否 lock。
7. GT RX reset done 是否为 1。
8. PCS block lock 是否为 1。
9. local fault / remote fault 是否存在。
```

典型原因：

```text
光纤收发接反
模块不兼容
参考时钟频率错
GT 复位顺序错
极性反了
对端不是 10GBASE-R
```

## block lock 失败

block lock 是 PCS 是否正确识别 66bit block 的关键状态。

常见原因：

```text
GT 没锁定
线路误码高
参考时钟错误
对端编码不是 64b/66b
RX polarity 错
复位释放过早
```

调试：

```text
看 GT PRBS 测试
看 block lock 状态是否抖动
看 high BER
尝试 polarity inversion
确认对端模式
```

## FCS 错误

FCS 错误说明 MAC 收到坏帧。

可能原因：

```text
物理误码
PCS/PMA 异常
AXI-Stream 数据被破坏
自定义 MAC/packet engine 生成错误
最后一拍 tkeep/tlast 错
```

如果是外部抓包看到 FCS 错误，先确认抓包设备是否真的保留 FCS；很多网卡不会把 FCS 交给上层。

## AXI-Stream 问题

重点看：

```text
tvalid && tready 才传输
tdata 在握手前必须保持稳定
tlast 标记包结束
tkeep 标记最后一拍有效字节
tuser 是否表示错误
```

ILA 触发建议：

```text
tvalid && !tready 持续过久
tlast 出现
tuser error
packet length counter overflow
FIFO almost full
FIFO underflow/overflow
```

## DMA 卡死

常见原因：

```text
descriptor owner 没切换
descriptor 地址错
buffer 未对齐
长度为 0 或超过限制
tlast 没到
中断没清
cache 没 flush
AXI bus error
```

排查：

```text
读 DMA status
读当前 descriptor 指针
检查 descriptor ring 内存
确认 tlast 是否到达 DMA
确认 AXI interconnect 没 decode error
```

## 丢包

丢包可能发生在多个位置：

```text
光口误码
MAC 丢弃错误帧
AXI FIFO overflow
DMA RX buffer 不足
DDR 写不及时
Firmware 处理太慢
上位机 socket buffer 太小
```

定位思路：

```text
比较各层计数器：
    MAC rx good packets
    MAC rx bad packets
    FIFO overflow
    DMA completed packets
    Firmware processed packets
    Application received packets
```

哪个计数开始少，问题就在那一层附近。

## 吞吐不足

常见瓶颈：

```text
包太小，pps 压力高
DMA descriptor 太少
中断太频繁
cache 操作开销大
DDR 带宽不足
AXI 总线带宽不足
上位机接收能力不足
TCP 窗口太小
没有启用 jumbo frame
```

优化：

```text
使用 jumbo frame
增加 DMA ring 深度
批量处理中断
减少拷贝
使用 zero-copy
增大 socket buffer
使用多队列/多流
提高应用处理线程优先级
```

## TCP 吞吐优化

关注：

```text
TCP window
MSS
RTT
丢包率
重传
拥塞控制
CPU 占用
```

10GbE TCP 要接近线速，通常需要：

```text
大窗口
大 buffer
低丢包
足够 CPU
checksum offload 或硬件辅助
jumbo frame
```

## UDP 吞吐优化

UDP 常见瓶颈是接收端丢包。

建议：

```text
使用 sequence number 统计丢包
控制发送速率
增大接收 buffer
使用 jumbo frame
减少每包中断
按批处理
```

## ILA 调试点

建议插 ILA：

```text
MAC AXI-Stream RX
MAC AXI-Stream TX
DMA S2MM input
DMA MM2S output
FIFO status
reset/status signals
PCS block lock
error counters
```

典型触发条件：

```text
tuser error
FIFO overflow
DMA error
block lock drop
tvalid stuck
tready stuck low
unexpected tlast
```

## 上位机抓包

常用工具：

```text
Wireshark
tcpdump
iperf3
ethtool
netstat/ss
```

过滤：

```text
eth
vlan
ip
udp
tcp
tcp.analysis.retransmission
udp.port == 5001
tcp.port == 5201
```

iperf3：

```bash
iperf3 -s
iperf3 -c <server_ip> -P 4
iperf3 -c <server_ip> -u -b 9G
```

注意：

```text
单条 TCP 流不一定跑满 10G。
多流并发常更容易接近线速。
UDP 测试要看丢包率，不只看发送速率。
```

## 最小排障口诀

```text
先物理：SFP+、LOS、refclk。
再 GT：PLL lock、reset done、CDR。
再 PCS：block lock、fault、BER。
再 MAC：FCS、length、pause、统计。
再 AXIS：tvalid/tready/tlast/tkeep。
再 DMA：descriptor、buffer、cache。
再应用：协议、socket、窗口、CPU。
```
