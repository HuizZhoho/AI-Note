# 13 从零到项目跑通逐步实操解读

## 本章解决什么问题

前面的文档按知识点和项目类型展开。本章换成“你真正动手时怎么一步步做”的视角，把以太网芯片项目拆成可执行步骤：

```text
先看硬件
  |
再读芯片 ID
  |
再让 link up
  |
再确认 MAC 帧收发
  |
再跑 ARP / ping
  |
再跑 UDP / TCP
  |
最后做应用和压力测试
```

每一步都回答四个问题：

```text
本步目标是什么？
具体做什么？
做成的证据是什么？
失败后往哪一层退？
```

## 总体方法：不要跳层

以太网项目最忌讳一上来就改应用代码。正确顺序是：

```text
Step 0：明确项目结构
Step 1：检查原理图和硬件接口
Step 2：确认电源、时钟、reset
Step 3：读芯片 ID
Step 4：让 PHY / SerDes link up
Step 5：确认 MAC 层收发计数
Step 6：跑 ARP
Step 7：跑 ICMP ping
Step 8：跑 UDP 或 TCP
Step 9：接真实应用
Step 10：做压力、异常和现场排障验证
```

如果某一步没有证据，不要进入下一步。

## Step 0：画出自己的项目结构

### 目标

先明确自己的板子到底是哪一种以太网架构。

### 操作

在纸上或文档里画出类似框图：

```text
PC
  |
Switch
  |
RJ45 / SFP+
  |
PHY / SerDes
  |
MAC
  |
DMA / FIFO
  |
CPU / FPGA / Firmware
  |
协议栈 / 应用
```

然后标出：

```text
速率：100M / 1000M / 10G
接口：RMII / RGMII / SGMII / 10GBASE-R / PCIe / AXI-Stream
管理：MDIO / I2C / SPI / PCIe BAR / AXI-Lite
软件：裸机 lwIP / Linux / FPGA Firmware / 自定义 RTL
应用：ping / UDP / TCP / Modbus TCP / 采集上报
```

### 成功证据

你能清楚说出：

```text
PHY 在哪里。
MAC 在哪里。
谁负责 DMA。
谁负责 TCP/IP。
哪个接口最容易出问题。
```

### 失败转向

如果说不清 PHY、MAC、Switch、NIC 的边界，先回到：

- [01 PHY 芯片硬件与驱动详解](./01_PHY芯片硬件与驱动详解.md)
- [02 Switch 交换芯片架构与配置](./02_Switch交换芯片架构与配置.md)
- [03 NIC 网卡芯片架构与驱动](./03_NIC网卡芯片架构与驱动.md)

## Step 1：检查原理图

### 目标

确认硬件连接和软件配置是否有可能匹配。

### 操作

逐项检查：

```text
PHY / Switch / NIC 型号。
MAC-PHY 接口类型。
管理接口地址。
reset 由谁控制。
参考时钟从哪里来。
IO 电压是否匹配。
RJ45 或 SFP+ 连接是否完整。
LED 和中断脚是否接出。
```

对 ZYNQ 千兆 PHY 项目，重点看：

```text
GEM0 还是 GEM1。
MIO 还是 EMIO。
PHY 地址 strap。
RGMII delay 模式。
PHY reset GPIO。
```

对 Switch 项目，重点看：

```text
CPU Port 接口。
用户端口编号。
每个端口是内置 PHY 还是外置 PHY。
管理 CPU 通过什么总线访问 Switch。
```

对 PCIe NIC 项目，重点看：

```text
PCIe REFCLK。
PERST#。
lane 宽度和极性。
EEPROM / Flash。
MAC 地址来源。
```

### 成功证据

你能写出一张表：

| 项目 | 实际值 |
| --- | --- |
| MAC-PHY 接口 | RGMII / SGMII / 10GBASE-R |
| PHY 地址 | 例如 1，按原理图确认 |
| Reset GPIO | 例如 MIOxx 或 PL GPIO |
| 参考时钟 | 25MHz / 50MHz / 125MHz / 156.25MHz |
| 软件接口 | lwIP / Linux DSA / NIC driver / Firmware |

### 失败转向

如果原理图信息缺失，不要写死参数。文档和代码中保留“按实际原理图确认”。

## Step 2：确认电源、时钟、reset

### 目标

让芯片处于能工作的基本状态。

### 操作

硬件上检查：

```text
电源电压是否正确。
reset 是否按 datasheet 要求释放。
参考时钟是否存在。
PHY / Switch / NIC 是否异常发热。
LED 是否符合预期。
```

软件上检查：

```text
reset GPIO 是否配置为输出。
reset 释放后是否等待足够时间。
时钟控制寄存器是否打开。
```

伪代码：

```c
static void board_eth_reset_sequence(void)
{
    eth_reset_assert();
    delay_ms(10);
    eth_reset_deassert();
    delay_ms(100);
}
```

### 成功证据

```text
示波器看到 reset 释放。
示波器看到参考时钟。
芯片不异常发热。
后续管理接口能读 ID。
```

### 失败转向

如果本步失败，不要继续看 MDIO、ARP、TCP。先修硬件电气。

## Step 3：读芯片 ID

### 目标

证明 CPU、Firmware 或驱动能访问目标芯片。

### 操作

PHY：

```text
通过 MDIO 读 PHY ID1 / ID2。
扫描 0 到 31 号 PHY 地址。
```

Switch：

```text
通过 MDIO / SPI / I2C / PCIe / SDK 读 chip ID。
```

NIC：

```bash
lspci -nn | grep -i ethernet
lspci -vv -s <bus:dev.fn>
```

FPGA 自定义 IP：

```text
通过 AXI-Lite 读 version register。
```

### 成功证据

```text
PHY ID 不是 0xffff 或 0x0000。
Switch chip ID 与型号匹配。
PCIe NIC 能枚举出 vendor/device ID。
FPGA IP version register 可读。
```

### 失败转向

```text
PHY ID 读不到：
    回查 MDIO、PHY 地址、reset、时钟。

Switch ID 读不到：
    回查管理总线、strap、reset、SDK 初始化。

PCIe 枚举不到：
    回查 REFCLK、PERST#、lane、BIOS/设备树。
```

## Step 4：让 link up

### 目标

证明物理链路已经建立。

### 操作

铜口 PHY：

```text
插网线。
读 link status。
读 speed / duplex。
确认 auto-negotiation complete。
```

Linux：

```bash
ethtool eth0
ethtool lan0
```

10G FPGA：

```text
检查 GT PLL lock。
检查 PCS block lock。
检查 hi_ber 是否为 0。
检查 SFP+ LOS 和 TX_DISABLE。
```

### 成功证据

```text
link up。
速率符合预期。
双工模式符合预期。
Switch 对端端口也显示 up。
```

### 失败转向

```text
link 不 up：
    查网线、模块、对端、速率、自协商。

100M up，1000M 不 up：
    查 RGMII delay、时钟、PCB、电平。

10G block lock 失败：
    查 GT refclk、lane 极性、SFP+、PCS/PMA reset。
```

## Step 5：确认 MAC 层收发

### 目标

证明以太网帧能进出 MAC，而不只是 link 灯亮。

### 操作

查看计数：

```text
MAC RX frame count。
MAC TX frame count。
FCS / CRC error。
RX dropped。
TX error。
DMA descriptor 状态。
```

Linux：

```bash
ip -s link show eth0
ethtool -S eth0
```

FPGA：

```text
ILA 观察 AXI-Stream：
tvalid、tready、tlast、tkeep、tuser。
```

### 成功证据

```text
发包时 TX 计数增加。
收包时 RX 计数增加。
FCS/CRC error 不持续增加。
DMA descriptor 能正常回收。
```

### 失败转向

```text
link up 但 RX 不增加：
    查 MAC 是否 enable、端口/VLAN、PHY-MAC 接口。

RX 增加但 FCS 错：
    查 RGMII/SGMII/10G PCS 时序和信号质量。

DMA 不回收：
    查 descriptor、cache、tlast、buffer 对齐。
```

## Step 6：跑 ARP

### 目标

证明二层广播和 IPv4 地址解析正常。

### 操作

PC：

```powershell
arp -d *
ping 192.168.1.10
arp -a
```

抓包过滤：

```text
arp
```

预期：

```text
PC -> ff:ff:ff:ff:ff:ff：Who has 192.168.1.10?
Device -> PC：192.168.1.10 is at xx:xx:xx:xx:xx:xx
```

### 成功证据

```text
PC ARP 表出现设备 IP 与 MAC。
抓包看到 ARP reply。
Switch MAC 表学习到 PC 和设备 MAC。
```

### 失败转向

```text
ARP request 到不了设备：
    查 Switch VLAN、PVID、STP、MAC RX。

设备收到但不回：
    查 IP、netif、ARP 处理、xemacif_input。

设备回了 PC 收不到：
    查交换机转发、VLAN、PC 网卡、防火墙。
```

## Step 7：跑 ping

### 目标

证明 IPv4 和 ICMP 正常。

### 操作

```powershell
ping 192.168.1.10
```

抓包过滤：

```text
icmp
```

### 成功证据

```text
ICMP Echo Request。
ICMP Echo Reply。
往返延迟稳定。
无明显丢包。
```

### 失败转向

```text
ARP 通但 ping 不通：
    查 IP 配置、防火墙、ICMP 是否启用、FCS 错误、ACL。
```

## Step 8：跑 UDP 或 TCP

### 目标

证明传输层和应用入口正常。

### UDP 操作

适合先验证数据面：

```text
设备启动 UDP echo。
PC 用网络调试助手发送 UDP。
设备原样返回。
```

检查：

```text
udp_recv 是否进入。
pbuf 是否释放。
udp_sendto 是否返回成功。
```

### TCP 操作

适合验证完整连接：

```powershell
Test-NetConnection 192.168.1.10 -Port 7
```

或：

```text
TCP Client -> 192.168.1.10:7
发送字符串
设备返回相同字符串
```

检查调用链：

```text
tcp_new_ip_type()
tcp_bind()
tcp_listen()
tcp_accept()
tcp_recv()
recv_callback()
tcp_write()
tcp_output()
```

### 成功证据

```text
UDP 能收能回。
TCP 有 SYN、SYN ACK、ACK。
应用 callback 进入。
应用数据正确返回。
```

### 失败转向

```text
ping 通但 TCP 不通：
    查端口、防火墙、listen、tcp_accept。

TCP 能连但不回数据：
    查 recv_callback、tcp_write、tcp_output、pbuf_free。

大数据卡住：
    查 tcp_sndbuf、tcp_sent、TCP 窗口、lwIP 内存。
```

## Step 9：接真实应用

### 目标

把网络链路接到你的业务。

典型应用：

```text
Modbus TCP 控制。
UDP 数据采集。
TCP 文件或状态上传。
Web 配置页面。
工业设备心跳。
```

### 操作

先定义应用协议：

```text
消息头。
长度。
命令字。
序号。
payload。
校验或状态码。
超时和重传策略。
```

UDP 采集建议：

```text
sequence。
timestamp。
payload_len。
stream_id。
```

TCP 控制建议：

```text
命令响应。
超时。
断线重连。
状态机。
错误码。
```

### 成功证据

```text
应用能处理正常请求。
异常请求不会导致死机。
断线后能恢复。
统计计数能说明收了多少、发了多少、丢了多少。
```

## Step 10：压力和异常验证

### 目标

证明项目不是只在实验室偶尔跑通。

### 操作

逐项验证：

```text
长时间 ping。
iperf 吞吐。
UDP 高速连续发送。
插拔网线。
重启交换机。
PC 端口 down/up。
错误 IP。
错误端口。
VLAN 配错。
大包、Jumbo frame。
多连接。
```

Linux 工具：

```bash
iperf3 -c <ip> -P 4
tcpdump -i eth0 -nn -e
ethtool -S eth0
```

ZYNQ/FPGA 关注：

```text
DMA busy。
pbuf 内存池。
cache flush/invalidate。
FIFO overflow。
AXI-Stream tready 背压。
FCS error。
```

### 成功证据

```text
长时间运行不死机。
断线恢复后可重新通信。
错误计数可解释。
吞吐接近设计目标。
日志能定位故障层级。
```

## 五个案例的逐步学习路径

### 路径 A：ZYNQ 千兆 PHY

```text
1. 看原理图：GEM0/GEM1、MIO/EMIO、PHY 地址、reset。
2. Vivado 打开 GEM 和 MDIO。
3. Vitis 创建 lwIP echo server。
4. 串口确认 xemac_add 成功。
5. 读 PHY ID 和 link。
6. PC ping。
7. TCP echo。
8. 抓包解释 ARP、ICMP、TCP。
```

### 路径 B：Switch 网关

```text
1. 读 Switch chip ID。
2. 确认 CPU Port link。
3. 确认 user port link。
4. 配 VLAN 和 PVID。
5. 看 MAC 表学习。
6. 同 VLAN ping。
7. 不同 VLAN 隔离。
8. 看 CPU trap 和端口统计。
```

### 路径 C：PCIe NIC

```text
1. lspci 确认枚举。
2. 驱动 probe 读 BAR。
3. 初始化 RX/TX ring。
4. link up。
5. ping。
6. iperf。
7. ethtool -S 看丢包和错误。
8. 调整 ring、RSS、offload。
```

### 路径 D：FPGA 10G UDP

```text
1. SFP+ 模块和 GT refclk 正常。
2. GT PLL lock。
3. PCS block lock。
4. MAC TX/RX 计数。
5. AXI-Stream tvalid/tready/tlast 正常。
6. PC 抓到 UDP 包。
7. 检查 sequence 是否连续。
8. 提升速率并观察 FIFO overflow。
```

### 路径 E：工业 Modbus TCP

```text
1. 静态 IP 配置。
2. ping 设备。
3. TCP 502 端口连接。
4. 读 Holding Registers。
5. 写 Single Register。
6. 异常功能码测试。
7. 断线重连。
8. 长时间运行和现场日志验证。
```

## 最终掌握标准

当你能做到下面这些，就说明真正掌握了以太网芯片项目：

```text
看到原理图能判断 PHY/MAC/Switch/NIC 的边界。
能解释 reset、时钟、strap、MDIO、RGMII/SGMII 的作用。
能从 PHY ID、link、MAC 计数、ARP、ping、TCP 分层定位问题。
能看懂 DMA descriptor、pbuf、AXI-Stream、NAPI 的数据搬运逻辑。
能用抓包解释一次完整通信过程。
能把一个示例迁移到自己的板卡，而不是只改 IP 地址。
```

一句话总结：

```text
以太网项目不是从应用开始调，而是从硬件证据开始，一层一层往上证明。
```
