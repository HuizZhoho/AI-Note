# 09 交换机内部实现

## 本章解决什么问题

前面的章节讲交换机“表现出来”的行为。本章从内部实现看交换机为什么能高速转发：

```text
端口接收
帧解析
分类
MAC/VLAN 查表
输出端口向量生成
交换矩阵或共享内存转发
队列和调度
端口发送
```

## 交换机内部总流程

```text
Port RX
  |
PHY/MAC 接收
  |
帧校验
  |
解析 Ethernet/VLAN/优先级
  |
分类引擎
  |
学习源 MAC
  |
查目的 MAC
  |
生成输出端口集合
  |
进入交换 fabric
  |
输出队列
  |
调度
  |
Port TX
```

## 接收路径

接收路径要做：

```text
检查帧长度。
检查 FCS。
识别 VLAN tag。
识别保留组播地址。
提取源 MAC、目的 MAC、EtherType。
根据规则分类。
```

保留地址可能需要本地处理，例如：

```text
STP BPDU。
LACP。
LLDP。
PAUSE。
```

这些帧不一定按普通数据帧转发。

## 分类引擎

分类引擎决定这帧属于哪类流量。

可能依据：

```text
入端口
VLAN ID
源/目的 MAC
EtherType
IP 地址
TCP/UDP 端口
DSCP/PCP
ACL 规则
```

分类结果用于：

```text
安全过滤。
QoS 队列选择。
镜像。
策略转发。
统计计数。
```

## 查表

二层查表常见键：

```text
VLAN ID + MAC Address
```

输出：

```text
出口端口或端口集合。
是否需要泛洪。
是否本地处理。
是否丢弃。
```

实现方式：

```text
CAM：
    高速精确匹配。

TCAM：
    支持通配匹配，常用于 ACL。

哈希表：
    成本较低，但要处理冲突。
```

## 交换结构

### 共享内存

所有端口共享一块高速缓冲。

优点：

```text
缓冲利用率高。
组播复制相对灵活。
```

问题：

```text
内存带宽要求高。
高速多端口场景压力大。
```

### 共享总线

多个端口共享内部总线。

优点：

```text
结构简单。
适合较低速或早期设计。
```

问题：

```text
总线容易成为瓶颈。
扩展性有限。
```

### 交叉矩阵

输入和输出之间通过 crossbar 连接。

优点：

```text
并行度高。
适合高速多端口。
```

问题：

```text
调度复杂。
可能出现 Head-of-Line Blocking。
```

## HOL Blocking

HOL 是 Head-of-Line Blocking。

通俗理解：

```text
队伍最前面的包要去一个拥塞出口。
它堵住了后面本来可以去其他空闲出口的包。
```

结果：

```text
即使某些出口空闲，也可能因为输入队列头部被堵而无法发送。
```

缓解方式：

```text
虚拟输出队列 VOQ。
更复杂的调度算法。
增加内部带宽。
合理队列设计。
```

## 队列设计

队列可能存在于：

```text
输入端口。
输出端口。
共享内存。
每个优先级。
每个流或每个服务类别。
```

输出队列常用于 QoS：

```text
高优先级队列。
普通队列。
低优先级队列。
控制协议队列。
```

调度器决定谁先发：

```text
严格优先级。
加权轮询。
赤字轮询。
整形和限速。
```

## 组播复制

组播和广播需要复制到多个出口。

实现位置可能是：

```text
入口复制：
    入方向生成多个副本。

fabric 复制：
    在交换结构中复制。

出口复制：
    到出口侧再复制。
```

不同实现影响：

```text
内部带宽消耗。
缓冲占用。
延迟。
芯片复杂度。
```

## 为什么交换芯片会有资源限制

交换机不是无限能力设备。常见资源：

```text
MAC 表容量。
ARP/ND 表容量。
ACL TCAM 容量。
路由表容量。
组播表容量。
队列数量。
缓冲大小。
镜像会话数量。
```

资源耗尽会导致：

```text
未知单播泛洪增加。
ACL 无法下发到硬件。
路由走软件慢路径。
丢包增加。
CPU 升高。
```

## 现代网络实践

现代交换机还会涉及：

```text
SDN/OpenFlow 或可编程 pipeline。
VXLAN/EVPN。
Telemetry。
PFC/ECN 数据中心拥塞控制。
白盒交换机。
P4 可编程转发。
```

但无论技术如何演进，基础路径仍然是：

```text
解析 -> 分类 -> 查表 -> 排队 -> 调度 -> 转发
```

## 排障视角

如果怀疑内部资源或芯片层问题：

```text
查看硬件转发表是否满。
查看 TCAM 使用率。
查看队列丢包。
查看 CPU punt 流量。
查看是否有报文走软件转发。
查看端口 buffer 是否拥塞。
```

典型现象：

```text
配置看似正确但流量异常：
    可能规则未下发硬件。

只有某类流量慢：
    查分类、队列、ACL、QoS。

大规模 MAC 学习异常：
    查 MAC 表容量和老化。
```
