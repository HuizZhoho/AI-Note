# 02 透明桥与交换机原理

## 本章解决什么问题

本章解释交换机最核心的行为：

```text
学习源 MAC。
查找目的 MAC。
已知则转发。
未知则泛洪。
同端口则过滤。
长期不用则老化。
```

所谓透明桥，是指终端主机不需要知道桥或交换机的存在。主机仍然像在同一个以太网里通信，桥在中间自动学习和转发。

## 初学者通俗理解

交换机像一个逐渐熟悉住户位置的门卫。

```text
第一次看到 A 从 1 号门进来：
    记下 A 在 1 号门。

A 要找 B：
    如果不知道 B 在哪，就广播问所有门。

后来 B 从 3 号门回话：
    记下 B 在 3 号门。

以后 A 找 B：
    只把帧发到 3 号门，不打扰其他门。
```

这就是交换机比集线器高效的原因。

## 核心机制

### 学习

收到一帧后，交换机先看源 MAC：

```text
源 MAC = A
入口端口 = port 1
VLAN = 10
```

于是记录：

```text
(VLAN 10, MAC A) -> port 1
```

注意：真实交换机的 MAC 表通常是按 VLAN 分开的。同一个 MAC 在不同 VLAN 中可能有不同上下文。

### 查找

交换机再看目的 MAC：

```text
目的 MAC = B
```

如果表中有：

```text
(VLAN 10, MAC B) -> port 3
```

就把帧转发到 port 3。

### 泛洪

如果查不到目的 MAC，交换机会在同一个 VLAN 内向除入口端口外的其他端口发送。

泛洪对象包括：

```text
未知单播
广播
部分组播
```

泛洪不是错误，而是以太网工作机制的一部分。问题在于环路环境下，泛洪会被无限放大。

### 过滤

如果目的 MAC 查到的出口端口和入口端口相同，交换机会丢弃这帧。

原因：

```text
目的主机和源主机已经在同一端口方向。
没有必要再从同一端口发回去。
```

### 老化

MAC 表项不是永久的。设备移动、网线变化、虚拟机迁移都会让 MAC 位置变化。

所以交换机会设置 aging timer：

```text
一段时间没看到某 MAC 的帧
  |
删除该 MAC 表项
  |
下次再通信时重新学习
```

## MAC 表实现

常见实现方式：

```text
软件表：
    便宜、灵活，但速度较慢。

哈希表：
    查找快，可能有冲突处理。

CAM/TCAM：
    硬件并行查找，非常快，常用于交换芯片。
```

CAM 的直观理解：

```text
普通内存：
    给地址，返回内容。

CAM：
    给内容，返回它在哪。
```

交换机需要线速查找 MAC，所以硬件查表能力很关键。

## 转发流程示意

```text
收到帧
  |
检查 FCS/长度
  |
学习源 MAC + VLAN + 入端口
  |
查找目的 MAC + VLAN
  |
已知单播 -> 指定端口
未知单播 -> 同 VLAN 泛洪
广播 -> 同 VLAN 泛洪
组播 -> 查组播表或泛洪
  |
输出排队
  |
发送
```

## 交换机为什么不是多口网线

集线器的行为：

```text
一个端口收到信号，复制到所有端口。
所有设备共享带宽。
冲突域很大。
```

交换机的行为：

```text
每个端口独立收发。
根据 MAC 表选择性转发。
多个端口可同时通信。
每个端口通常是独立冲突域。
```

现代全双工交换以太网基本不再依赖 CSMA/CD 冲突检测。

## 常见误区

```text
误区 1：交换机一开始就知道所有 MAC。
    交换机需要通过源 MAC 学习。

误区 2：未知单播一定代表故障。
    刚开始通信、表项老化、主机移动时都会有未知单播。

误区 3：MAC 表越大越好。
    表太小会频繁泛洪，太大则增加成本和功耗。需要匹配场景。

误区 4：广播只发给一个端口。
    广播会在同 VLAN 内泛洪。
```

## 排障视角

常用检查：

```text
查看 MAC 地址表：
    是否能看到目标 MAC。

查看 VLAN：
    MAC 是否在正确 VLAN 中学习。

查看端口状态：
    link、速率、双工、错误包。

查看泛洪：
    是否存在未知单播过多。
```

典型问题：

```text
MAC 表学不到：
    端口没 up、VLAN 错、设备没发包、链路异常。

MAC 在端口之间来回跳：
    可能有环路、双网卡桥接、虚拟化迁移或配置错误。

大量未知单播：
    MAC 表溢出、老化时间太短、单向流量、攻击。
```
