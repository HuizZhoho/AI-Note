# 06 组播、优先级、QoS

## 本章解决什么问题

本章解释三类进阶二层能力：

```text
组播：
    一份数据发给一组接收者。

优先级：
    不同流量有不同重要程度。

QoS：
    通过分类、标记、排队、调度和丢弃策略改善关键业务体验。
```

## 组播

### 通俗理解

单播像给一个人发消息，广播像在全楼广播，组播像只给订阅某个频道的人发消息。

```text
单播：
    one to one

广播：
    one to all

组播：
    one to many selected receivers
```

### 二层组播问题

如果交换机不知道哪些端口有组播接收者，就可能像广播一样泛洪组播流量。

这会造成：

```text
无关端口收到大量视频/音频流。
带宽浪费。
终端 CPU 被无关组播打扰。
```

### 组播裁剪

组播裁剪的目标：

```text
只把组播流量转发到有接收者的端口。
```

常见机制：

```text
IGMP Snooping：
    交换机监听主机和路由器之间的 IGMP 报文，建立组播转发表。

MLD Snooping：
    IPv6 组播场景中监听 MLD。
```

老式 LAN 技术中也有 GMRP 等注册机制，但现代以太网部署更常见 IGMP/MLD Snooping。

## 优先级

### 为什么需要优先级

链路拥塞时，不同业务不应完全同等对待。

例如：

```text
语音：
    延迟敏感。

视频会议：
    延迟和抖动敏感。

文件下载：
    吞吐敏感，但可等待。

备份：
    可低优先级。
```

### 802.1p

802.1Q Tag 中有 PCP 字段，3 bit，可以表示 8 个优先级：

```text
0 - 7
```

PCP 是二层优先级标记，经常用于交换机内部队列选择。

## QoS

QoS 不是凭空增加带宽，而是在拥塞时决定：

```text
谁先发。
谁后发。
谁可以被丢。
谁要保证低延迟。
```

### QoS 处理流程

```text
流量进入交换机
  |
分类 Classify
  |
标记 Mark
  |
排队 Queue
  |
调度 Schedule
  |
必要时丢弃 Drop
  |
发送
```

### 分类依据

可能依据：

```text
端口
VLAN
802.1p PCP
DSCP
源/目的 IP
TCP/UDP 端口
ACL 匹配结果
```

### 队列和调度

常见调度：

```text
Strict Priority：
    高优先级队列先发，低优先级可能被饿死。

Weighted Round Robin：
    按权重轮询多个队列。

Deficit Round Robin：
    改进不同包长下的公平性。
```

工程上常组合使用：

```text
语音高优先级。
普通业务正常队列。
备份和下载低优先级。
```

## CoS 和 QoS

```text
CoS：
    Class of Service，更偏分类和优先级标记。

QoS：
    Quality of Service，更完整，包括分类、标记、排队、调度、整形、监管等。
```

## 常见误区

```text
误区 1：QoS 能让 1G 链路变成 2G。
    QoS 不能增加物理带宽，只能在拥塞时分配资源。

误区 2：优先级越高越好。
    如果所有流量都标高优先级，等于没有优先级。

误区 3：组播天然只到接收者。
    交换机需要组播监听或注册机制，否则可能泛洪。

误区 4：丢包一定是坏事。
    拥塞时合理丢弃低优先级流量，可以保护关键业务。
```

## 排障视角

组播问题：

```text
确认 IGMP/MLD Snooping 是否启用。
确认是否有 Querier。
查看组播转发表。
抓 IGMP Join/Leave。
```

QoS 问题：

```text
查看入方向标记是否正确。
查看队列映射。
查看队列丢包计数。
确认调度策略。
确认 DSCP/PCP 是否被中间设备重写。
```

Wireshark 观察：

```text
eth.dst[0] & 1 == 1
igmp
icmpv6
vlan.priority
ip.dsfield.dscp
```
